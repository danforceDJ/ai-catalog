# User Stories — Company AI Catalog Marketplace

> **Inspired by:** [github/awesome-copilot](https://github.com/github/awesome-copilot)  
> **Target repo:** [danforceDJ/ai-catalog](https://github.com/danforceDJ/ai-catalog)  
> **Date created:** 2026-04-21

---

## Background & Analysis

`github/awesome-copilot` is a **community-curated, open-contribution** marketplace for GitHub Copilot customizations. Its structure:

| Concept | awesome-copilot | ai-catalog (company) |
|---|---|---|
| Content types | agents, instructions, skills, plugins, hooks, agentic workflows | skills, agents, prompts (commands), MCP servers, templates, bundles |
| Directory layout | flat top-level (`agents/`, `skills/`, `instructions/`, `hooks/`) | **flat top-level** (`skills/`, `agents/`, `commands/`, `mcpServers/`) + bundle manifests in `plugins/` |
| Discoverability | public website + llms.txt + npm build pipeline | GitHub Pages + Copilot CLI plugin format |
| Install method | `copilot plugin install <name>@awesome-copilot` | `copilot plugin install <name>@ai-catalog` |
| Contribution model | open community PRs | internal team PRs (company-controlled) |
| Toolchain | Node.js / npm (`npm run build`) | Python / uv (PEP 723 scripts) |
| Marketplace manifest | `.github/plugin/marketplace.json` (generated by npm) | `.github/plugin/marketplace.json` (generated by Python) |
| Validation | `npm run plugin:validate` + `npm run skill:validate` | `uv run --script scripts/validate_catalog.py` |
| CI | GitHub Actions (validate + build + deploy) | GitHub Actions (validate) + GitLab CI (validate + generate) |

**Key simplification opportunities** identified by comparing the two repos:

1. ~~`awesome-copilot` has flat top-level folders (`agents/`, `skills/`, `instructions/`). `ai-catalog` nests everything under `plugins/<name>/` — this makes each plugin self-contained and easier to install as a unit, which is the right call for a company catalog.~~ **Resolved:** `ai-catalog` now uses a flat top-level layout (`skills/`, `agents/`, `commands/`, `mcpServers/`). Plugins are pure bundle manifests that reference primitives by name list. Primitives are shareable across bundles without duplication.
2. `awesome-copilot` includes `hooks/` and `workflows/` types not yet present in `ai-catalog` — these are high-value for internal CI/DevOps automation.
3. `awesome-copilot` provides a `Learning Hub` and `Cookbook` for onboarding; `ai-catalog` has no equivalent onboarding material yet.
4. `awesome-copilot` supports `instructions/` (auto-applied `.instructions.md` files) — not present in `ai-catalog` but directly applicable to company coding standards.

## Epic 1 — Catalog Discovery & Installation

### US-101 — Browse catalog via web marketplace

**As a** developer in the company,  
**I want to** open a searchable web page and browse all available AI tools (skills, agents, prompts, MCP servers),  
**so that** I can discover what the company has built without reading the source repo.

**Acceptance criteria:**
- A GitHub Pages site lists every plugin and template.
- Each card shows name, type, description, tags, and install options.
- Full-text search filters the visible cards in real time.
- The page is auto-generated on every push to `main` — no manual step.

---

### US-102 — Install a plugin via Copilot CLI

**As a** developer using GitHub Copilot CLI,  
**I want to** run a single command to install any skill, agent, prompt, or MCP server from the catalog,  
**so that** I can get started without copying files manually.

**Acceptance criteria:**
- `copilot plugin marketplace add danforceDJ/ai-catalog` registers the catalog once.
- `copilot plugin install <name>@ai-catalog` installs any plugin.
- Install command is shown on every card in the web marketplace.

---

### US-103 — Install an MCP server in VS Code via deeplink

**As a** developer using VS Code,  
**I want to** click a single button in the web marketplace to add an MCP server to my VS Code configuration,  
**so that** I don't have to edit JSON files manually.

**Acceptance criteria:**
- Every MCP plugin card has a "Install in VS Code" deeplink button.
- Clicking the button opens VS Code and merges the server config into `.vscode/mcp.json`.
- The deeplink is auto-generated from the plugin's `.mcp.json`.

---

### US-104 — Download a single file or zip bundle

**As a** developer who prefers manual installation,  
**I want to** copy the raw file content or download a zip bundle from the web marketplace,  
**so that** I can install catalog items without the Copilot CLI.

**Acceptance criteria:**
- Single-file items (skills, agents, prompts) show a "Copy raw file" button.
- All plugins offer a downloadable `.zip` bundle.
- Both options are available on every marketplace card.

## Epic 2 — Content Creation & Contribution

### US-201 — Add a new Agent Skill to the catalog

**As a** developer who has built a reusable AI skill,  
**I want to** add it to the company catalog by creating a folder with a `SKILL.md` file,  
**so that** my teammates can discover and install it.

**Acceptance criteria:**
- I create `skills/<skill-name>/SKILL.md` with frontmatter (`name`, `description`, `version`).
- To make it Copilot CLI installable, I optionally create `plugins/<name>/plugin.json` declaring `"skills": ["<skill-name>"]`.
- Running `uv run --script scripts/validate_catalog.py` passes with no errors.
- After my PR is merged, the skill appears in the web marketplace and is installable via CLI (via its plugin wrapper).

---

### US-202 — Add a new Agent Profile to the catalog

**As a** platform engineer,  
**I want to** publish a company-standard agent profile (AGENTS.md-compatible) to the catalog,  
**so that** all developers get consistent agent behavior (commit conventions, security rules, etc.) without manual setup.

**Acceptance criteria:**
- I create `agents/<agent-name>.agent.md` with frontmatter.
- To make it Copilot CLI installable, I create `plugins/<name>/plugin.json` declaring `"agents": ["<agent-name>"]`.
- Validation passes; the agent appears in the marketplace after merge.

---

### US-203 — Add a new Prompt (Slash Command) to the catalog

**As a** developer,  
**I want to** publish a reusable slash command (e.g., `/pr-description`, `/commit-message`),  
**so that** my teammates can use it in Copilot Chat without re-typing the same prompt.

**Acceptance criteria:**
- I create `commands/<command-name>.md` with frontmatter.
- Filename stem matches the `name` field in frontmatter (enforced by validation).
- To make it Copilot CLI installable, I create `plugins/<name>/plugin.json` declaring `"commands": ["<command-name>"]`.
- Command appears in the marketplace after merge.

---

### US-204 — Add an MCP Server configuration to the catalog

**As a** platform engineer,  
**I want to** publish an MCP server configuration (e.g., for Atlassian, Jira, Slack),  
**so that** developers can connect Copilot to internal tools with a single install command.

**Acceptance criteria:**
- I create `mcpServers/<name>/.mcp.json` in the Copilot `{"servers": {...}}` shape.
- To make it Copilot CLI installable, I create `plugins/<name>/plugin.json` declaring `"mcpServers": ["<name>"]`.
- No secrets or tokens are hard-coded (OAuth/SSO or environment variable references only).
- Validation scans for secret patterns and fails if found.
- MCP plugin appears in the marketplace with a VS Code deeplink.

---

### US-205 — Add a reusable document Template to the catalog

**As a** developer,  
**I want to** publish a reusable Markdown document template (e.g., Solution Architecture, RFC, ADR),  
**so that** my teammates can download a ready-to-fill starting point.

**Acceptance criteria:**
- I create `templates/<name>/TEMPLATE.md` with frontmatter (`name`, `description`, `version`, `category`).
- Template appears in the web marketplace with download-only affordances (no Copilot CLI install).
- Template is available as a zip download.

---

### US-206 — Add a Bundle Plugin (skill + MCP + agent)

**As a** platform team,  
**I want to** publish a bundle plugin that groups a skill, MCP server, and agent profile into a single installable unit for a specific workflow (e.g., "Atlassian value-stream bundle"),  
**so that** developers get everything they need in one install command.

**Acceptance criteria:**
- Plugin `plugin.json` declares ≥2 component types using name lists (e.g., `"skills": ["a", "b"]`, `"mcpServers": ["x"]`).
- Referenced primitives exist in their top-level directories (`skills/`, `agents/`, `commands/`, `mcpServers/`).
- `copilot plugin install <bundle-name>@ai-catalog` installs all components.
- Bundle appears in the marketplace with a distinct "bundle" badge and can be filtered by type "bundle".
- **Example:** `plugins/value-stream-1-default/` bundles `skills/publish-to-confluence`, `skills/generate-architecture-doc`, `skills/jira-ticket-from-code`, and `mcpServers/atlassian`. ✅ Live in catalog.

---

### US-207 — Add coding standard Instructions (auto-applied)

**As a** tech lead,  
**I want to** publish company coding standards as `.instructions.md` files (à la `awesome-copilot`),  
**so that** Copilot automatically applies them to files matching a given pattern (e.g., `**.ts`, `**.py`) without any developer action.

**Acceptance criteria:**
- A new top-level `instructions/<name>/` folder supports `.instructions.md` files (follows the same flat layout as `skills/`, `agents/`, etc.).
- Each file has `applyTo` and `description` frontmatter fields.
- Instructions are discoverable in the web marketplace and can be installed via CLI.
- Validation enforces required frontmatter fields.

---

### US-208 — Add an Agentic Hook to the catalog

**As a** DevOps engineer,  
**I want to** publish automated hooks that run during Copilot agent sessions (e.g., lint on save, test on code change),  
**so that** agent-assisted development enforces quality gates automatically.

**Acceptance criteria:**
- A new top-level `hooks/<name>/` folder supports hook definitions (follows the flat layout convention).
- Hook has a `hooks.json` configuration and a `README.md` with frontmatter.
- Validation checks required fields.
- Hooks appear in the marketplace and are installable.

## Epic 3 — Validation & Quality

### US-301 — Validate catalog on every PR

**As a** catalog maintainer,  
**I want to** have automated validation run on every pull request,  
**so that** invalid plugin manifests, duplicate names, or committed secrets never reach `main`.

**Acceptance criteria:**
- GitHub Actions runs `validate_catalog.py` on every PR.
- CI fails with a clear error message if validation fails.
- CI checks that `catalog.json` and `.github/plugin/marketplace.json` are up to date.

---

### US-302 — Prevent secrets from being committed

**As a** security engineer,  
**I want to** have the validation script scan all `.mcp.json` files for common secret patterns,  
**so that** API tokens and credentials are never accidentally published to the catalog.

**Acceptance criteria:**
- `validate_catalog.py` scans for patterns like `ghp_`, `API_TOKEN=`, `SECRET=`, etc.
- Any match causes validation to fail with the file path and matched pattern shown.
- CI blocks the PR until the secret is removed.

---

### US-303 — Auto-regenerate catalog artifacts on commit

**As a** contributor,  
**I want to** have `catalog.json` and `.github/plugin/marketplace.json` regenerated automatically when I commit plugin changes,  
**so that** I never accidentally push stale artifacts and break CI.

**Acceptance criteria:**
- A pre-commit hook runs `generate_catalog.py` and `generate_marketplace.py` when `plugins/`, `skills/`, `agents/`, `commands/`, `mcpServers/`, or `templates/` files change.
- The hook auto-stages the regenerated files.
- If regeneration fails (e.g., invalid plugin), the commit is blocked with an error.

---

## Epic 4 — Developer Experience & Onboarding

### US-401 — Onboarding guide for new contributors

**As a** new developer joining the company,  
**I want to** follow a clear, step-by-step guide to add my first plugin to the catalog,  
**so that** I don't need to read all the scripts to understand how things work.

**Acceptance criteria:**
- `CONTRIBUTING.md` explains every component type (skill, agent, prompt, MCP, template) with a concrete example.
- A "Local development setup" section explains pre-commit hook installation.
- `README.md` has a "Quick Start" section with the 3 most common CLI commands.

---

### US-402 — Scaffolding command for new plugins

**As a** developer starting a new plugin,  
**I want to** run a single command to scaffold the folder structure for a new plugin,  
**so that** I don't have to look up the required file names and frontmatter fields.

**Acceptance criteria:**
- A script or Copilot CLI command creates the boilerplate directory structure for a given plugin type.
- Generated files contain placeholder frontmatter with field descriptions.
- Running validation on a freshly scaffolded (but otherwise empty) plugin passes with no errors.

---

### US-403 — Machine-readable catalog for AI agents (llms.txt)

**As a** developer building an AI agent that needs to discover company tools,  
**I want to** have a structured `llms.txt` (or equivalent) listing all catalog items,  
**so that** AI agents can programmatically find and install the right tools without scraping HTML.

**Acceptance criteria:**
- A `llms.txt` file (or JSON equivalent) is generated at build time and published to GitHub Pages.
- It lists all plugins and templates with name, type, description, and install command.
- It is linked from `README.md`.

---

### US-404 — Categorize and tag catalog items

**As a** developer browsing the marketplace,  
**I want to** filter catalog items by category (e.g., `integrations`, `productivity`, `governance`) and tags,  
**so that** I can find relevant tools quickly in a large catalog.

**Acceptance criteria:**
- Every plugin has `category` and `tags` fields in `plugin.json`.
- The web marketplace renders category and tag filters.
- Filtering by category shows only matching cards.

---

## Epic 5 — Governance & Platform Consistency

### US-501 — Enforce naming conventions across the catalog

**As a** catalog maintainer,  
**I want to** have validation enforce naming conventions (kebab-case, ≤64 chars, directory name matches `plugin.json` `name`),  
**so that** all catalog items are consistently named and Copilot CLI can always find them.

**Acceptance criteria:**
- Validation fails if `plugin.json` `name` does not match the directory name.
- Validation fails if `name` is not kebab-case or exceeds 64 characters.
- Validation fails if a command file's stem does not match its frontmatter `name`.

---

### US-502 — Prevent duplicate plugin or command names

**As a** catalog maintainer,  
**I want to** have validation detect duplicate plugin names and MCP server names across the catalog,  
**so that** there are no naming collisions that would break `copilot plugin install`.

**Acceptance criteria:**
- Validation fails with a clear error if two plugins share the same `name`.
- Validation fails if two MCP server names collide across different plugins.
- Validation fails if two commands share the same `name`.

---

### US-503 — Support GitLab CI alongside GitHub Actions

**As a** developer working in a GitLab-mirrored environment,  
**I want to** have the same validate-and-generate pipeline run in GitLab CI,  
**so that** the catalog is always valid regardless of which git platform is used.

**Acceptance criteria:**
- `.gitlab-ci.yml` runs validation and artifact generation on every MR.
- Bash wrapper scripts delegate to Python scripts (no logic duplication).
- GitLab pipeline passes on a clean repo without any changes to `.gitlab-ci.yml`.
